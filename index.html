Goal in. Harness plans. Crabbox starts a worker. Airbyte moves rows. Evidence returns.

This is the exact mechanism. The harness is the driver. It builds a job spec and calls Crabbox. Crabbox leases a sandbox worker with scoped credentials. Inside that worker, Airbyte reads from the source and writes to the target. The harness receives proof, not the row stream.

1 / tools

What tools exist and what each one does.

Keep these roles separate. If one component starts doing every job, the architecture becomes hard to audit and unsafe around credentials.

Harness

The driver. It can be an agent graph, scheduler, backend service, or CLI wrapper.

  • Reads objective and state.
  • Creates job spec.
  • Calls crabbox.
  • Reads evidence and decides next step.

Crabbox

The execution substrate. It makes a remote box available and runs the command there.

  • pool ensure keeps workers warm.
  • run --pool leases a sandbox.
  • Profiles and allowlists scope credentials.
  • History/results/artifacts return proof.

Sandbox worker

The process boundary that touches data. It is short-lived or borrowed from a ready pool.

  • Repo is hydrated.
  • Airbyte and scripts run here.
  • Secrets materialize only here.
  • Reports and metrics are written here.

Airbyte

The data mover inside the sandbox. It handles connector logic and source-to-target movement.

  • Reads source records.
  • Writes target records.
  • Uses connector config generated by the harness.
  • Emits sync status for validation.

Source and target

External systems. Examples: SaaS API, Postgres, S3, warehouse, CRM, or operational app.

  • Source is read by Airbyte.
  • Target is written by Airbyte.
  • Harness stores references, not row streams.

Evidence

The feedback layer. It is what the harness reads after execution.

  • Logs and phase events.
  • Row counts and timings.
  • JUnit and validation reports.
  • Artifacts and redacted config.
2 / 14-step trace

The full run, end to end.

This is the actual sequence. It names who is active, what they call, and what changes hands.

01

Objective enters the harness.

A human, scheduler, webhook, or data-change event asks for a data outcome.

input: goal + policy
02

Harness reads state.

It checks warehouse freshness, previous Crabbox history, existing artifacts, schema state, and destination constraints.

crabbox history --json
03

Harness chooses the movement type.

It decides whether this is standard ETL, reverse ETL, enrichment, validation, repair, or a mixed job.

decision: worker kind
04

Harness chooses source and target references.

It names the systems and target config. It does not fetch and hold the data rows.

source_ref + target_ref
05

Harness chooses the credential profile.

It selects a profile such as etl-warehouse. Raw secret values stay in profiles or the secret store.

--env-from-profile
06

Harness builds the job spec.

The spec contains pool, Airbyte connector config, script, env allowlist, artifact globs, validation rules, and retry policy. It is written into the workspace that Crabbox syncs to the worker.

.crabbox/generated/job.json
07

Harness ensures capacity.

It asks Crabbox to keep the right pool warm so dispatch is fast.

crabbox pool ensure
08

Harness starts the run.

It calls Crabbox with the pool, command, env allowlist, profile, artifact rules, and JUnit path.

crabbox run --pool
09

Crabbox leases the sandbox.

Crabbox borrows or creates a worker, hydrates the repo, attaches cache volumes, and injects allowed environment values.

worker lease
10

Worker starts the sync script.

The script reads the synced generated job file, prepares Airbyte config, and starts the connector or PyAirbyte flow.

python -m workers.sync
11

Airbyte reads the source.

Airbyte uses worker-scoped credentials to pull records from the source. This happens inside the sandbox.

source -> worker
12

Airbyte writes the target.

Records are normalized, streamed, loaded, or emitted according to the connector and target config.

worker -> target
13

Worker validates and packages evidence.

It emits phases, row counts, timings, validation reports, JUnit, logs, redacted config, and artifacts.

reports/** metrics.json
14

Harness reads evidence and decides next action.

The harness finishes, retries, alerts, opens a follow-up job, or changes the next plan.

results + artifacts
3 / worker cutaway

What runs inside the sandbox worker.

The worker is not just a terminal. It is the execution envelope around Airbyte, generated config, scoped credentials, validation, and artifacts.

sandbox-worker inside the box

Airbyte runs here

Airbyte is installed or hydrated in the worker environment, then invoked by a script or PyAirbyte wrapper.

Config is generated before execution

The harness creates or selects connector config, but secrets are represented by profile references and allowed environment variables.

Validation is local to the worker

The worker can compare row counts, check schema drift, write JUnit, and save redacted config for audit.

4 / paths

Three paths exist. Do not mix them.

This is the most important design rule. Control, data, and evidence are separate flows.

Control path

caller -> harness -> Crabbox -> worker command. Contains job specs, profile names, config references, retry rules, and status.

Data path

source -> Airbyte in worker -> target. Contains rows and records. The harness should not stream or store this path.

Evidence path

worker -> Crabbox history/results/artifacts -> harness. Contains logs, metrics, JUnit, reports, samples, and redacted config.

5 / credentials

How credentials work.

"Harness spins up Crabbox with creds and target" means profile references and target config. Raw secrets should not be in prompts, command lines, logs, or artifact files.

Harness chooses

credential_profile = etl-warehouse, allow_env = AIRBYTE_*,SOURCE_*,TARGET_*.

Crabbox injects

Crabbox resolves profile-backed values and exposes only allowed environment variables to the sandbox process.

Worker uses

Airbyte uses those values to authenticate source and target. Reports must redact secrets.

Credential rule

The harness may decide which credential profile is appropriate. It should not become the secret store. Secrets materialize at worker execution time and should not come back in evidence.

6 / runnable shape

The calls look like this.

This is the minimal shell shape a harness can emit. A real harness can call a local CLI, wrapper service, or future API, but the boundary is the same.

harness-dispatch.sh 
# 1. Harness keeps the right worker type warm.
crabbox pool ensure example-org/data-movement/main/provider/linux/etl \
  --min-ready 3 \
  --create -- \
  --cache-volume airbyte-etl

# 2. Harness writes the job spec.
# It contains connector config, source ref, target ref, validation rules.
# It references a credential profile; it does not contain raw secrets.
mkdir -p .crabbox/generated
cat > .crabbox/generated/accounts-sync.json <<'JSON'
{
  "movement": "source_to_target",
  "source_ref": "source.crm.accounts",
  "target_ref": "warehouse.analytics.accounts",
  "airbyte_connection": "accounts_sync",
  "validation": ["row_count", "schema_drift", "freshness"],
  "redact": ["password", "token", "secret"]
}
JSON

# 3. Harness dispatches the worker run through Crabbox.
crabbox run --pool example-org/data-movement/main/provider/linux/etl \
  --shell 'python -m workers.airbyte_sync --config .crabbox/generated/accounts-sync.json' \
  --allow-env 'AIRBYTE_*,SOURCE_*,TARGET_*,WAREHOUSE_*' \
  --env-from-profile etl-warehouse \
  --artifact-glob 'reports/**,metrics.json,redacted-config.json' \
  --junit reports/

# 4. Harness reads evidence and decides the next step.
crabbox history --limit 50 --json
crabbox results <run-id> --json
crabbox artifacts download <run-id> --out evidence/<run-id>

What calls what

The harness calls Crabbox. Crabbox starts the worker command. The worker command invokes Airbyte. Airbyte calls source and target systems. The harness reads Crabbox results and artifacts.

7 / rationale

Why this design is worth doing.

The goal is not to add layers. The goal is to keep intelligence dynamic while making execution bounded, auditable, and repeatable.

Security

The harness plans with references. The worker executes with scoped secrets. Secrets stay out of prompts and logs.

Observability

Every run produces structured proof: phases, metrics, JUnit, artifacts, and history.

Reproducibility

The worker runtime is repo-defined and hydrated consistently before execution.

Adaptability

The harness can change the plan next run without giving up execution control.
