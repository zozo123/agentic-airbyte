THE CORE NOTION

Using Crabbox Remote Sandboxes + Airbyte for a New Kind of ETL/rETL

Traditional ETL and reverse ETL are static pipelines defined by humans in UIs or YAML and run on fixed schedules. They break easily, can't adapt, and have almost no place for AI agents to participate meaningfully.

The core notion: A new version of ETL and rETL where AI agents don't just plan — they control and execute data movement inside secure remote sandboxes powered by Crabbox, using Airbyte for the reliable connectors.

How it works intuitively:

  • ETL (forward): Agent leases a Crabbox remote sandbox (warm pooled box). Inside it, runs Airbyte to extract from sources (DBs, SaaS, APIs) and load to warehouse/dest. Or generates/runs custom transforms in Python.
  • rETL (reverse): Agent uses the sandbox to read from warehouse, applies AI-driven logic (enrichment, personalization, eligibility), and pushes to operational systems (CRMs, apps, on-prem) — all safely isolated.

Crabbox provides the remote sandboxes: on-demand or pre-warmed isolated environments (via pools/prewarm from docs) with your full stack hydrated via Actions (Airbyte CLI + CDK + Python). Secrets forwarded safely, every run produces artifacts and audit for the agent to observe and iterate.

Airbyte handles the standard, reliable "pipes". Crabbox remote sandboxes give agents the power and safety to do the dynamic, custom, reverse, and complex parts that make this a truly new, agentic version of ETL/rETL.

Old Way (Not Agentic)

  • • Static configs in Airbyte UI or YAML
  • • Scheduled by cron or orchestrator — no intelligence
  • • Custom logic is painful (separate scripts, no isolation, no audit)
  • • Agents can only suggest changes; humans still do the work
  • • No closed loop for the AI to observe results and adapt

New Way (Agentic with Remote Sandboxes)

  • • Agent decides what to move and how (including generating custom code)
  • • Uses Crabbox to get a real remote sandbox on demand or from a warm pool
  • • Runs Airbyte + custom logic safely inside that sandbox
  • • Gets structured results back (artifacts, metrics) to reason over
  • • Full audit trail of every AI-driven data movement decision
  • • Fast iteration because sandboxes can be pre-warmed and reused
THE TWO PIECES

The Tools

Airbyte

THE RELIABLE CONNECTORS

The open-source standard for moving data between sources and destinations with hundreds of connectors and excellent observability.

  • 300+ connectors, CDK for customs
  • CLI, API, and Python libraries agents can drive
  • Great for the standard, declarative part of movement

Crabbox

THE REMOTE SANDBOX LAYER

Gives agents (and humans) on-demand or pre-warmed remote execution environments ("sandboxes" or "boxes") with full control, safety, and observability. Perfect place to run Airbyte or custom ETL code.

Remote Sandboxes on Demand — Pools + prewarm give you warm, reusable isolated Linux environments ready for data work.
Safe Execution — Strict env allowlisting for secrets, per-lease isolation, no secrets in broker.
Hydration from your repo — Use your existing GitHub Actions workflows to set up Airbyte + Python + whatever inside the sandbox.
Full Observability for Agents — Artifacts, results, phases, and history so the agent can see what happened and decide what to do next.

Step 1: Get a Remote Sandbox
Use crabbox pool ensure <key> --min-ready 1 --create (or prewarm). Crabbox provisions or reuses an isolated remote environment. Use actions hydration so the sandbox already has Airbyte installed exactly as in your CI.
This is the "remote sandbox" part — the agent doesn't run things locally or on some untrusted code interpreter. It gets a real, controlled remote box.
Step 2: Agent Decides What to Do
The agent looks at current state (warehouse, Airbyte connections, previous runs). It can decide to trigger a standard Airbyte sync, generate a custom CDK connector, run a complex transform, or do reverse ETL with business logic.
Step 3: Execute Inside the Sandbox
Agent (via its tools) calls something like:
crabbox run --pool <key> --shell 'airbyte ...' --allow-env 'AIRBYTE_*' ...
or --script my_agent_generated_etl.py with proper artifact capture.
Everything sensitive or custom happens inside the Crabbox-provided remote sandbox with safe secret forwarding and full output capture.
Step 4: Observe and Close the Loop
Agent gets back artifacts (data files, reports), JUnit metrics, logs, and phases. It can now reason about success/failure/drift and decide the next action (including reverse ETL steps). Every step leaves a complete, auditable record.

How to Actually Use This

1. Set up a data-ops repo with Crabbox + Airbyte
Add crabbox.yaml or .crabbox.yaml. Point actions.workflow at a GitHub Actions file that installs Airbyte CLI, Python, Airbyte CDK, etc. Use cache volumes for deps.
2. Pre-warm / ensure remote sandbox capacity
crabbox pool ensure my-data-pipelines/main/hetzner/linux/cpx51 --min-ready 2 --create -- --provider hetzner --type cpx51 --cache-volume airbyte-etl
This creates warm remote sandboxes ready for agent work.
3. Give your agent Crabbox as a tool
In your agent system prompt or tool registry, include the ability to run crabbox commands (pool, run --pool, etc.). The agent will discover it can lease remote sandboxes and run Airbyte or custom code inside them.
4. Let the agent drive the flow
Example pattern the agent will produce:
crabbox run --pool ... --shell 'airbyte connections create ...' --allow-env 'AIRBYTE_*'
or generate + run a Python script for custom logic with --script, --artifact-glob, and proper env forwarding.
Key safety & power features from Crabbox (directly from the docs):
• Warm pooled remote sandboxes so agents aren't waiting for cold VMs
• Actions hydration so the sandbox has exactly the Airbyte + data environment your repo expects
• Strict environment allowlisting so agents can only forward the secrets they actually need
• Artifacts + results + history so every data movement the agent triggers is observable and auditable
