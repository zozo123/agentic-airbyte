How & Why This Is New for ETL & rETL

Traditional data movement was declarative and static. Agentic data movement is dynamic, reasoning-driven, and execution-safe.

Traditional Limitations

  • • Static & brittle: Pipelines defined in UIs or YAML. Schema changes or new sources require manual intervention.
  • • Limited intelligence: No reasoning loop. Can't discover sources, generate custom logic, or decide transformations based on live data.
  • • Execution gaps: Custom or reverse logic lives in throwaway scripts with poor isolation, no audit, and cold starts.
  • • Poor AI integration: Agents can plan but can't safely act on real data movement at scale.
  • • Siloed audit: No unified trace of "why the agent decided to move this data this way."

Why This Pattern Is New

  • • Agentic loop for data: The first pattern that closes the observe-plan-execute-reflect cycle for ETL/rETL using production-grade tools.
  • • AI generates the "how": Agents don't just configure — they write custom Airbyte CDK connectors, transforms, and reverse logic on the fly and test them safely.
  • • Execution substrate for agents: Crabbox gives agents warm, isolated, auditable "hands" (pools, hydration, env forwarding, artifacts) — something no data platform provided before.
  • • True bidirectional + custom: Airbyte for the 80% standard movement + Crabbox for the long tail (on-prem, complex AI manipulation, regulated reverse) in one unified agent-driven system.
  • • Full provenance: Every AI decision and data movement is a first-class, reproducible Crabbox run record (see docs on history, artifacts, results).
This is the shift from "data pipelines as code" to "data movement as an agentic capability."

Crabbox

THE EXECUTION LAYER FOR AGENTS

Remote execution control plane. Gives agents warm, isolated, auditable hands.

Pools + Prewarm (docs/commands/pool.md, prewarm.md)
Reusable warm boxes. Register hydrated leases. Borrow in seconds. No cold starts for agents.
Actions Hydration (docs/features/actions-hydration.md)
Run your repo's .github/workflows to install Airbyte CLI, CDK, Python, deps — exact CI parity inside the box.
Env Forwarding + Profiles (docs/features/env-forwarding.md)
Strict allowlist for AIRBYTE_*, WAREHOUSE_*, DEST_*. Never leak secrets into agent prompts or code.
Artifacts, Results, History (docs/features/artifacts.md, commands/results.md)
--artifact-glob, --junit, phases, full run records. The agent can observe and audit every movement.
Cache Volumes + Jobs (docs/features/cache-volumes.md, features/jobs.md)
Persistent deps for heavy data tools. Named, repeatable multi-step agentic flows.

The Novelty

This is the first practical pattern that makes data movement a first-class agentic capability instead of a separate static infrastructure concern. Airbyte provides the commodity pipes at scale. Crabbox (drawing from real features like pools, prewarm, actions-hydration, env-forwarding, and artifacts documented in the project) gives agents the safe, fast, observable "body" they need to act on real data without hallucinating pipelines or leaking secrets. The result: ETL and rETL that can discover, customize, reverse, and iterate under AI control — with complete provenance.

IN PRACTICE

Real agent command surface

# Capacity (from pool.md + prewarm)
crabbox pool ensure ... --min-ready 2 --create -- --provider ... --cache-volume airbyte-etl

# Agent configures standard forward via audited shell
crabbox run --pool ... --shell 'airbyte connections create ...' --allow-env 'AIRBYTE_*'

# Agent runs custom / reverse (generated CDK or script)
crabbox run --pool ... --script /tmp/agent_reverse.py \

  --env-from-profile ~/.agent/creds.env \
