Exact execution model

The harness drives. Crabbox runs the worker. Airbyte moves the data.

This page explains the actual mechanism. The harness receives a goal, creates a concrete job spec, and calls Crabbox. Crabbox leases an isolated sandbox. Inside that sandbox, Airbyte reads from the source and writes to the target. The harness gets evidence back, not the data stream.

See the model Source
Driver The harness is the driver. It decides what should run and calls Crabbox.
Execution Crabbox leases or creates a sandbox worker, hydrates the repo, injects allowed environment values, and runs the command.
Data path Data flows source → sandbox worker running Airbyte → destination. It does not flow through the harness.
Feedback Logs, phases, metrics, JUnit, and artifacts flow back to Crabbox and then to the harness.
1 / model

Separate control from data movement.

The harness controls the job. The worker touches the data. Crabbox is the execution boundary between them.

system-flow control != data
Control calls: harness → Crabbox → worker.
Data path: source → Airbyte in worker → target.
Evidence path: worker → Crabbox → harness.
2 / trace

What happens, step by step.

This is the call sequence. No magic. The harness is the driver until Crabbox starts the worker command. Then the worker owns the data movement.

01

Caller gives the harness an objective.

Example: keep account records fresh in the warehouse and sync qualified accounts to a target application.

goal = {source, target, freshness, policy, schedule}
02

Harness turns the objective into a job spec.

The spec names the worker pool, Airbyte connector/config, source reference, target reference, credential profile, artifact rules, and validation checks.

job = {pool, airbyte_config, target_config, credential_profile, allow_env, artifacts}
03

Harness calls Crabbox.

It may first ensure capacity, then it starts a run against the chosen pool. The harness passes config references and profile names, not raw secrets.

crabbox pool ensure ... && crabbox run --pool ... --env-from-profile ...
04

Crabbox leases the sandbox worker.

Crabbox borrows or creates a remote sandbox, hydrates the repository, attaches cache volumes if configured, injects only allowed environment values, and starts the command.

worker.exec("python -m workers.airbyte_sync --config generated.json")
05

Airbyte runs inside the worker.

Airbyte, PyAirbyte, or connector code reads from the source using worker-scoped credentials and writes to the target. This is the data plane.

source → worker/Airbyte → target
06

Worker validates and emits evidence.

The worker writes phases, row counts, status, logs, JUnit, validation reports, and artifacts. Sensitive values are redacted or omitted.

reports/**, metrics.json, junit.xml, CRABBOX_PHASE events
07

Harness reads the evidence and decides.

The harness calls Crabbox history/results/artifacts, updates state, and chooses whether to finish, retry, alert, or launch a follow-up job.

crabbox results <run-id> --json
3 / data path

How the data moves.

There are two flows: control messages and data movement. Confusing them is the main source of bad architecture.

Control flow

caller → harness → Crabbox → sandbox command. This flow contains job specs, config references, profile names, and status.

Data flow

source → Airbyte in sandbox → target. The harness does not stream rows. The Crabbox broker does not transform rows.

Evidence flow

sandbox → Crabbox results/artifacts → harness. This flow contains metrics, logs, test results, samples, and reports.

4 / credentials

How credentials are handled.

The harness can decide which credential profile to use. It should not carry raw secrets in prompts, plans, command lines, or logs.

  • The harness names a profile. Example: etl-warehouse or activation-safe.
  • Crabbox resolves allowed environment values. Use --env-from-profile and --allow-env.
  • The worker receives the secrets at execution time. Only the sandbox process sees them.
  • Outputs are evidence, not secret dumps. Reports should include counts, statuses, and redacted config, not credentials.

Important distinction

"Harness spins up Crabbox with creds and target" should mean: the harness passes a target config plus credential profile reference. Raw secret values should stay in profiles or the secret store and only materialize inside the worker.

5 / calls

What calls what.

The harness can be an agent service, scheduler, LangGraph workflow, or normal backend. The calls still look like this.

harness-dispatch.sh 
# Harness ensures the worker pool exists and stays warm.
crabbox pool ensure example-org/data-movement/main/provider/linux/etl \
  --min-ready 3 \
  --create -- \
  --cache-volume airbyte-etl

# Harness dispatches one data movement job.
# It passes target config and a credential profile reference.
# Raw secrets are not embedded in the command.
crabbox run --pool example-org/data-movement/main/provider/linux/etl \
  --shell 'python -m workers.airbyte_sync --config /tmp/job.json' \
  --allow-env 'AIRBYTE_*,SOURCE_*,TARGET_*,WAREHOUSE_*' \
  --env-from-profile etl-warehouse \
  --artifact-glob 'reports/**,metrics.json,redacted-config.json' \
  --junit reports/

# Harness reads evidence and decides the next action.
crabbox history --limit 50 --json
crabbox results <run-id> --json
crabbox artifacts download <run-id> --out evidence/<run-id>

Who is the driver?

The harness is the driver. Crabbox is the execution substrate. Airbyte is the data mover inside the sandbox. The source and target are external systems. Evidence is the feedback loop.

6 / rationale

Why design it this way.

The design exists to let the harness make dynamic decisions without making the harness the place where data, secrets, and mutable execution state accumulate.

Bounded blast radius

Each job runs in a sandbox with scoped credentials and a clear artifact boundary.

Real execution

Airbyte and code run in an actual repo-defined environment, not inside a vague agent tool abstraction.

Useful memory

The harness learns from run evidence: what moved, what failed, what changed, and what should happen next.
