THE PROBLEM

Airbyte is great at moving data.
Agents are great at deciding how.

Static connections and scheduled syncs don't cut it anymore. Agents can (and should) dynamically configure pipelines, generate custom logic, and drive true bidirectional src ↔ dst — but they need a safe place to actually run the work.

THE AGENTIC PATTERN
1. OBSERVE
Agent reads Airbyte state, samples data, reviews past Crabbox runs.
2. MANIPULATE
AI decides config changes, writes custom CDK/transforms, plans reverse activation.
3. EXECUTE
Everything custom or sensitive runs via crabbox run --pool on warm, isolated boxes.
4. REFLECT
Agent ingests artifacts + results, updates its plan, loops.
Forward (src → dst)
Airbyte owns the standard 300+ connectors and reliable syncs (to warehouse or directly to destinations). Agents dynamically create/update connections via the Airbyte API and use Crabbox to run/test custom sources or transforms.
Reverse + Bidirectional (dst → src / activation)
The hard part lives in Crabbox: AI-driven reverse ETL, personalized pushes, complex eligibility, on-prem systems, and true src ↔ dst loops. Agents write the logic. Crabbox executes it safely with full audit.
IN PRACTICE

What an agent actually runs

# 1. Agent configures Airbyte (audited via Crabbox)
crabbox run --pool data-pipelines/main/... --shell \
  'airbyte connections create --source new-events --destination warehouse'

# 2. Agent generates + executes custom manipulation / reverse
cat > /tmp/ai_manip.py << 'EOF'
# AI-written: custom Airbyte CDK or heavy transform / reverse logic
print("CRABBOX_PHASE:ai-manipulation rows=18472")
EOF

crabbox run --pool data-pipelines/main/hetzner/linux/cpx51 \
  --script /tmp/ai_manip.py \
  --allow-env 'AIRBYTE_*,WAREHOUSE_*,DEST_*' \
  --env-from-profile ~/.agent/creds.env \
  --artifact-glob 'reports/*' \
  --junit manipulation.xml \
  --label "agent:airbyte-src-dst:$(date -I)"

# 3. Agent reflects on results + Airbyte state, loops
