Traditional Limitations

  • • Static & brittle: Defined in UIs/YAML. Changes require human intervention. No dynamic discovery or generation.
  • • No reasoning loop: Agents can suggest but can't safely execute custom logic, test in real envs, or iterate fast.
  • • Execution is afterthought: Custom/reverse lives in ad-hoc scripts with cold starts, secret leaks, no unified audit.
  • • Limited to platform: Hard to do complex AI-driven transforms, on-prem, or true bidirectional without separate fragile tools.

Why This Is Fundamentally New

  • • Closes the agentic loop for data: First pattern with observe-plan-execute-reflect using production tools (Airbyte for pipes, Crabbox for execution per docs on pools/prewarm/actions-hydration).
  • • AI owns the "how": Agents generate and test custom Airbyte CDK, transforms, reverse logic on-the-fly inside safe warm boxes — not limited to UI.
  • • Crabbox as execution substrate: Provides warm reusable workers (pool ensure + prewarm from docs), repo-native setup (actions-hydration), secret-safe forwarding (env-forwarding), full evidence (artifacts/results), for agents to act reliably.
  • • Unified bidirectional + audit: Standard via Airbyte + complex/reverse/on-prem via Crabbox in one auditable agent-driven system. Every AI decision has provenance (Crabbox history + artifacts).
  • • Fast & safe iteration: Seconds not minutes (warm pools + cache volumes). Isolation and allowlists prevent leaks. Reproducible via git + hydration.
This shifts ETL/rETL from "static infrastructure" to "agentic capability" — agents can now reliably discover, customize, execute, and govern data movement end-to-end.

How the Flow Works (Step by Step with Crabbox Docs)

1. Warm Reusable Workers (Pools + Prewarm)

Agent calls (or has tool for) crabbox pool ensure <key> --min-ready N --create -- --provider …. This creates or reuses hydrated leases (docs/commands/pool.md). Prewarm (docs/commands/prewarm.md) leases and runs actions hydrate from your repo workflow to install Airbyte + data tools. Result: boxes ready in seconds for any agent iteration.
Why: Traditional cold VMs or containers take minutes. Agents need fast loops. Pools give "always-on" workers with fingerprint matching to avoid poison (see ready pools in spec/broker.md).

2. Repo-Native Setup (Actions Hydration)

In crabbox.yaml or config: actions.workflow points to .github/workflows/airbyte-stack.yml. During first run or prewarm, Crabbox translates and runs it over SSH (local hydration) or as self-hosted runner. Installs airbyte, pip install airbyte, etc. Writes readiness marker. See docs/features/actions-hydration.md for local vs github-runner paths.
Why for agentic: Your data/Airbyte setup lives in git with the rest of the project. Agents get identical env to CI without the agent framework knowing about Airbyte install details.

3. Safe Execution of Agent Logic (Run + Env + Artifacts)

Agent generates script/config. Executes via crabbox run --pool <key> --script ... --allow-env 'AIRBYTE_*,... ' --env-from-profile ... --artifact-glob ... --junit .... Box sources hydrated env. Secrets forwarded only for allowed names (env-forwarding.md). Outputs (reports, CSVs from reverse) captured as artifacts. Full run record in history.
Why new: Agents can safely run both Airbyte commands and arbitrary custom/reverse logic (on-prem via SSH providers, complex transforms) with built-in isolation, secret hygiene, and evidence collection — no other tool gives agents this production execution surface for data movement.

4. Observation + Iteration (Artifacts + Jobs + History)

Agent downloads artifacts, parses JUnit for metrics, reads phases and history. Uses jobs (docs/features/jobs.md) for named multi-step flows. Cache volumes keep deps hot. Loop closes with full provenance.
Why for agents: The agent gets structured, machine-readable feedback (artifacts + results) to reason over, exactly like a human operator would — but automated and audited.
