The Vision

Instruct the harness. It owns the outcome via sandbox workers.

Traditional ETL/rETL is static pipelines. Raw agents with direct tool access are powerful but fragile on isolation, secrets, audit, and reproducibility.

The vision: You talk to a dedicated agent harness at the level of goals and policies. The harness plans, ensures capacity, and dispatches every actual movement to a fleet of Crabbox remote sandboxes that function as specialized, fully-auditable ETL and rETL workers.

You
"Keep Salesforce in sync with the latest warehouse profiles. Enrich with activity model. Respect PII policy X."
Harness (brain)
Observes state, plans ETL vs rETL, ensures worker pools, generates or selects logic, dispatches tasks, collects evidence, reflects, retries or alerts.
Sandbox Workers (hands)
Warm pooled remote environments. Run Airbyte for standard pipes or custom/AI code for complex movement. Return full artifacts + history. Nothing leaks.

The Execution Layer

Sandboxes make outstanding ETL/rETL workers

Warm & Fast
  • • pool ensure --min-ready N keeps hydrated boxes ready.
  • • crabbox run --pool … borrows in seconds (no cold boot).
  • • Prewarm + actions hydration gives you exact stack on every worker.
Isolated & Safe
  • • Every task runs in its own remote environment (or freshly borrowed one).
  • • Strict --allow-env 'AIRBYTE_*,DEST_*' + profiles (600 perms, redacted).
  • • Harness never holds long-lived secrets or mutable state from prior runs.
Hydrated Exactly
  • • Your repo's .github/workflows installs Airbyte, PyAirbyte/CDK, Python libs, dbt, whatever the worker needs.
  • • Cache volumes for fast pip/uv/connector installs across borrows.
  • • Git-defined. Reproducible on any provider (Hetzner, AWS, your Proxmox on-prem, etc.).
Observable by Design
  • • Every run produces history, logs, events, results (with CRABBOX_PHASE markers), JUnit summaries, and --artifact-glob tarballs.
  • • Perfect memory for the harness: row counts, timings, failure details, custom metrics.
  • • Full provenance and audit trail for compliance or replay.

Different pools for different worker personalities: bulk-ingest workers, low-latency activation workers, on-prem/internal workers, GPU transform workers, etc. The harness chooses the right pool for the job.

Inside an ETL worker
  1. 1. Borrowed warm box with Airbyte + connectors already present.
  2. 2. Harness (or generated config) drives Airbyte sync from source(s) into warehouse (or direct).
  3. 3. Optional post-steps: dbt, validation, custom transform in same env.
  4. 4. Writes artifacts (row counts, sample, proof) + emits CRABBOX_PHASE markers. Returns.
Inside an rETL worker
  1. 1. Same borrow from a (possibly different) pool tuned for activation.
  2. 2. Reads modeled data from warehouse (via profile creds).
  3. 3. Runs AI-driven logic (scoring model, eligibility, personalization, enrichment via external APIs) + custom dest logic (CDK or direct).
  4. 4. Pushes to operational systems. Full audit + artifacts back to harness for compliance or replay.
Getting Started

Build the harness

1. Define your worker pools in a data-movement repo
Add crabbox.yaml. Create a .github/workflows/hydrate.yml that installs Airbyte (or PyAirbyte), CDK, Python libs, etc. Use cache volumes. The harness (or you) runs:
crabbox pool ensure example-org/data-movement/main/hetzner/linux/cpx51 --min-ready 3 --create -- --provider hetzner --cache-volume airbyte-etl
2. The harness is ordinary code + Crabbox calls
A thin Python service, LangGraph agent, scheduler, or even a persistent Crabbox job running the loop. It observes, plans, calls the CLI (or future API), and feeds results back into its context.
# Minimal harness sketch (pseudocode)
while True:
    goal = receive_goal_or_schedule()
    state = observe_warehouse() + crabbox_history()
    plan = llm_plan(goal, state)          # ETL mix? which pool? custom script?
    crabbox("pool ensure … --min-ready …")
    run_id = crabbox("run --pool {plan.pool} --script {plan.script} --allow-env '…' --artifact-glob 'reports/**'")
    evidence = collect(crabbox_results(run_id), artifacts(run_id), history(run_id))
    llm_reflect(goal, evidence)
    # update memory, alert, or schedule next
3. Real dispatch commands the harness emits
# ETL worker (Airbyte inside)
crabbox run --pool … \
  --shell 'python -m workers.airbyte_ingest …' \
  --allow-env 'AIRBYTE_*,SOURCE_*,WAREHOUSE_*' \
  --env-from-profile etl-warehouse \
  --artifact-glob 'reports/**,metrics.json' --junit reports/

# rETL / AI worker
crabbox run --pool … --script workers/activate_scored.py \
  --allow-env 'DEST_*,MODEL_API_*' --artifact-glob 'evidence/**'
4. Leverage the primitives
  • • jobs: for named, repeatable worker templates ("customer-360-sync").
  • • Profiles for per-worker secret bundles + allowlists.
  • • Ponds for coordinated multi-box movements when needed.
  • • Full history/events/artifacts as the harness's perfect, queryable memory of every movement.
The harness stays lightweight and auditable because the heavy, secret-bearing, stateful work always happens inside short-lived or borrowed Crabbox sandboxes that were defined in your repo. You get agentic flexibility on top of reproducible, observable, governed execution.
