AI agent + Crabbox + Airbyte

Plan in the agent. Run in the sandbox. Move data with Airbyte. Repair from evidence.

The harness can be Claude Code, Codex, OpenCode, or another AI coding agent. It writes a bounded job spec and calls Crabbox. Crabbox runs the worker. Airbyte moves rows. Evidence decides the next step.

entities

Who does what.

Keep these boundaries clean. Most bad designs collapse two of these into one box.

DriverAI harness chooses job, target, profile, validation, and next action.
RunnerCrabbox leases worker, hydrates repo, injects allowed env, collects results.
MoverAirbyte runs in the worker and moves records from source to target.

Caller

Input: goal. Output: policy and constraints.

Agent

Input: state. Output: job spec and command.

Crabbox

Input: run request. Output: run id and artifacts.

Worker

Input: repo, env, config. Output: exit code and reports.

Airbyte

Input: connector config. Output: target writes and sync status.

Profile

Input: profile name. Output: scoped env in worker only.

Evidence

Input: logs and metrics. Output: finish, retry, repair, or alert.

flows

Four lanes. Do not mix them.

Controlhuman/schedule -> agent -> Crabbox -> worker command
Credentialsprofile store -> Crabbox -> worker env. Never prompt text.
Datasource -> Airbyte in worker -> target. Never through the agent.
Evidenceworker -> Crabbox results/artifacts -> agent decision.
failure map

Debug by owner, not by guessing.

Every failure needs an owner and a signal. The agent repairs only from evidence.

F1 Plan

Owner: agent. Signal: spec diff. Fix target, movement, or validation.

F2 Capacity

Owner: Crabbox. Signal: pool/run status. Fix pool or hydration.

F3 Credentials

Owner: profile. Signal: redacted env report. Fix profile mapping.

F4 Connector

Owner: Airbyte/source. Signal: connector logs. Fix API, schema, auth scope.

F5 Partial write

Owner: worker/target. Signal: target counts. Check idempotency before retry.

F6 Validation

Owner: next plan. Signal: JUnit and metrics. Compile a repair job.

runnable shape

The agent emits a bounded job.

ai-agent-dispatch.sh 
# Goal: sync CRM accounts into the warehouse safely.

crabbox pool ensure example-org/data-movement/main/provider/linux/etl \
  --min-ready 3 \
  --create -- \
  --cache-volume airbyte-etl

mkdir -p .crabbox/generated
cat > .crabbox/generated/accounts-sync.json <<'JSON'
{
  "movement": "source_to_target",
  "source_ref": "source.crm.accounts",
  "target_ref": "warehouse.analytics.accounts",
  "airbyte_connection": "accounts_sync",
  "credential_profile": "etl-warehouse",
  "allow_env": ["AIRBYTE_*", "SOURCE_*", "TARGET_*"],
  "idempotency_key": "accounts_sync:daily",
  "retry": {
    "max_attempts": 2,
    "when": ["rate_limit", "transient_network"]
  },
  "validation": ["row_count", "schema_drift", "freshness"],
  "artifacts": ["reports/**", "metrics.json", "redacted-config.json"],
  "redact": ["password", "token", "secret"]
}
JSON

crabbox run --pool example-org/data-movement/main/provider/linux/etl \
  --shell 'python -m workers.airbyte_sync --config .crabbox/generated/accounts-sync.json' \
  --allow-env 'AIRBYTE_*,SOURCE_*,TARGET_*' \
  --env-from-profile etl-warehouse \
  --artifact-glob 'reports/**,metrics.json,redacted-config.json' \
  --junit reports/

crabbox results <run-id> --json
crabbox artifacts download <run-id> --out evidence/<run-id>

Spec boundary

References, allowlists, retry, validation, artifacts. No raw secrets. No copied rows. No prompt transcript.

Debug rule

Start from run id, phase log, connector log, metrics, JUnit, redacted config. Change one bounded input. Rerun.

Invariant

The agent may choose the profile. It must not become the secret store, data mover, or artifact sanitizer.

AI agent drives. Crabbox executes. Airbyte moves data. Evidence returns.
