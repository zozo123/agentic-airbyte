01 / operating model

Agents reason. Sandboxes execute.

Raw tool access makes agents powerful but hard to govern. This model keeps intelligence in the harness and moves all mutable, secret-bearing work into short-lived remote workers.

Intent arrives

Schedules, events, or humans describe the desired outcome: enrich, sync, activate, validate, or repair.

Harness plans

It reads state, chooses ETL or rETL, selects a pool, and emits a concrete crabbox run.

Crabbox leases

Warm, repo-hydrated sandboxes are borrowed from capacity pools with strict environment allowlists.

Evidence returns

The harness receives metrics, JUnit, artifacts, phases, logs, and history to decide whether to retry or finish.

02 / architecture

A governed execution fabric for Airbyte and custom data work.

Provider-specific details stay behind Crabbox adapters. The harness only asks for a worker shape, a command, secret policy, and the evidence contract.

03 / closed loop

The harness keeps learning from worker evidence.

Each run returns a durable record. The next plan is based on current warehouse state, Crabbox history, artifacts, and the actual result of the previous attempt.

05 / runbook

The harness emits ordinary Crabbox commands.

No special magic is required. A service, scheduler, agent graph, or long-running job can assemble these commands from the current plan.

1. Define pools in repo config

Use crabbox.yaml and Actions hydration to install Airbyte, Python, CDK packages, validation tools, and cache volumes.

2. Keep access scoped

Forward only the environment values each worker needs through --allow-env and profile-backed secret bundles.

3. Make proof first-class

Every worker writes artifacts, phases, metrics, and JUnit summaries so the harness can reason from real execution.

06 / evidence

The memory layer is made of run output.

The harness does not need to guess what happened. It receives durable records that are useful for retry logic, compliance, debugging, and replay.

rows.moved live

Operational metrics

Replayable artifacts

Validation reports, generated configs, sample rows, connector output, and proof bundles are attached to the run.
