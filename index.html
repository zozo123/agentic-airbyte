The Core Shift

Instruct the harness.
It owns the outcome through real workers.

Static pipelines are brittle. Raw agents calling tools directly are powerful but leak secrets, state, and audit across every step.

The vision: A dedicated agent harness receives high-level goals and turns them into reliable data movement by dispatching work exclusively to a fleet of Crabbox remote sandboxes that function as specialized ETL and rETL workers.

Run Airbyte for standard movement or custom/AI code for complex activation. Return every artifact, metric, and audit record. Nothing leaks into the harness.

The Execution Layer

Why Crabbox sandboxes make the best ETL/rETL workers

Warm & Fast
Seconds, not minutes
  • • pool ensure --min-ready N keeps pre-hydrated boxes ready in the broker.
  • • crabbox run --pool … borrows instantly (no cold boot, no rsync if --no-sync).
  • • Prewarm + repo-owned Actions hydration = exact stack (Airbyte, CDK, libs) on every worker.
Isolated & Secret-Safe
Per-task blast radius
  • • Every task runs in its own remote environment (fresh or freshly borrowed).
  • • Strict --allow-env + --env-from-profile (600 perms, redacted in logs).
  • • Harness never sees or holds the actual secrets or intermediate state.
Exactly Hydrated
Git-defined, reproducible
  • • Your .github/workflows/hydrate.yml installs Airbyte, PyAirbyte/CDK, Python, cache volumes, whatever the worker needs.
  • • Same environment on Hetzner, AWS, your Proxmox on-prem boxes, or delegated providers.
  • • Cache volumes (lockfile-hashed) make repeated borrows fast.
Observable by Design
Perfect memory for the harness
  • • Every run emits history, events (with CRABBOX_PHASE), results (JUnit), and --artifact-glob tarballs.
  • • Row counts, timings, custom metrics, failure details — everything the harness needs to reflect and decide.
  • • Full provenance for compliance, replay, or debugging.

Different pools for different personalities. The harness picks the right one for bulk ETL, low-latency rETL, on-prem, or AI-heavy work.

Inside an ETL worker
  1. 1. Borrow warm box with Airbyte + connectors already installed via Actions.
  2. 2. Harness drives Airbyte (or PyAirbyte) from sources into warehouse or direct destination.
  3. 3. Optional same-env post-processing: dbt, validation, light transforms.
  4. 4. Emit phases, write artifacts (row counts, samples, proofs), return. Full record for harness.
Inside an rETL worker
  1. 1. Borrow from activation-tuned pool (same fast path).
  2. 2. Read modeled data from warehouse using profile-forwarded credentials.
  3. 3. Run AI-driven logic (scoring, eligibility, personalization, external enrichment) + custom dest (CDK or direct).
  4. 4. Push to operational systems. Return complete audit + artifacts for compliance and the next decision.
Making It Real

Build the harness

1. Define worker pools in your data-movement repo
Add crabbox.yaml. Create .github/workflows/hydrate.yml that installs Airbyte/PyAirbyte, CDK, Python, cache volumes. Then:
crabbox pool ensure example-org/data-movement/main/hetzner/linux/cpx51 --min-ready 3 --create -- --provider hetzner --cache-volume airbyte-etl
2. The harness is ordinary code that calls Crabbox
A thin service, LangGraph/Crew agent, scheduler, or even a long-running Crabbox job. It observes, plans, calls the CLI (or future API), and feeds rich results back into its context.
Minimal harness loop (pseudocode)
while True:
    goal = receive_goal_or_schedule()
    state = observe_warehouse() + crabbox.history()
    plan = llm_plan(goal, state)   # which pool? Airbyte or custom? script?

    crabbox("pool ensure … --min-ready …")
    id = crabbox("run --pool {plan.pool} --script {plan.script} \
      --allow-env '…' --env-from-profile {plan.profile} \
      --artifact-glob 'reports/**' --junit reports/")

    evidence = collect( crabbox.results(id), artifacts(id), history(id) )
    llm_reflect(goal, evidence)   # update memory, alert, or loop
3. Real commands the harness actually emits
# ETL worker (Airbyte inside the sandbox)
crabbox run --pool … \
  --shell 'python -m workers.airbyte_ingest …' \
  --allow-env 'AIRBYTE_*,SOURCE_*,WAREHOUSE_*' \
  --env-from-profile etl-warehouse \
  --artifact-glob 'reports/**,metrics.json' --junit reports/

# rETL / AI worker
crabbox run --pool … --script workers/activate_scored.py \
  --allow-env 'DEST_*,MODEL_API_*' --artifact-glob 'evidence/**'
4. Use the primitives the harness loves
  • • jobs: in config for named, versioned worker templates the harness can invoke by name.
  • • Profiles for exact per-worker env allowlists + secret bundles.
  • • Ponds when a movement needs coordinated peer sandboxes.
  • • Full history + events + artifacts = the harness's perfect, queryable source of truth.
The harness stays lightweight because the heavy, secret-bearing, stateful work always happens inside short-lived or borrowed Crabbox sandboxes defined in your git repo. You get true agentic flexibility on top of reproducible, observable, governed execution.
